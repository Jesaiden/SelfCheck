# 用例状态判定细则

## PASS
满足以下条件时可判 PASS：
- 有足够强证据证明该行为符合预期
- 证据和该用例直接对应
- 不依赖主观猜测

## FAIL
满足以下任一条件可判 FAIL：
- 实际执行结果错误
- 日志清楚显示失败
- 页面/接口/状态与预期不符
- 数据错误、崩溃、逻辑错乱

## BLOCKED
适用：
- 没有环境
- 没有账号/权限
- 没有依赖
- 没有核心输入材料
- 无法进行关键执行验证

## NEEDS_HUMAN_CHECK
适用：
- 视觉体验
- 交互手感
- 真机兼容
- 需要人工判断的业务合理性
- 当前代理没有能力确认的项目

## 推荐写法
### 好写法
- “缺少真机截图，当前只能判 NEEDS_HUMAN_CHECK”
- “npm run test:e2e 无法执行，主流程验证 BLOCKED”
- “保存后历史页回显正常，截图与日志一致，判 PASS”

### 坏写法
- “理论上应该没问题”
- “代码看起来是对的”
- “大概率可用”
