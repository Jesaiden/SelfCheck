# 证据规则

## 一、判断顺序
判定某条用例时，先问：
1. 是否存在直接对应的执行结果？
2. 是否存在日志或截图支撑？
3. 是否只是静态推断？
4. 这些证据是否足以支撑 PASS？

## 二、哪些证据最有价值
优先级从高到低：
1. 实际执行成功结果
2. 对应用例的截图 / 录屏 / 日志
3. 自动化测试报告
4. 部分运行结果
5. 静态代码推断

## 三、常见误判
### 误判 1：页面文件存在 = 页面通过
错误原因：页面存在不代表路由能进入、状态正确、交互成功

### 误判 2：有存储 API = 数据持久化通过
错误原因：还需要验证写入时机、读取时机、异常退出、覆盖逻辑

### 误判 3：命令写在 package.json = 测试通过
错误原因：需要看实际执行结果，而不是只看命令定义

### 误判 4：逻辑看上去正确 = 主流程通过
错误原因：主流程必须看真实闭环

## 四、对代理的硬要求
- 证据要尽量对应具体用例，不要泛泛写“有日志”
- 没有证据时，要写“缺少什么证据”
- 对 P0，用例和证据最好一一对应
