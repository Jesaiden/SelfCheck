# RUNBOOK：代理执行手册

本手册用于指导代理按稳定顺序完成一轮自检。

## 一、启动前
先收集 4 类材料：
1. 项目配置
2. 测试用例
3. 代码或实现
4. 证据与执行能力

若 1 或 2 缺失，先不要直接进入判定，必须先标注“输入不完整”。

## 二、推荐执行顺序
### Step 1：识别项目
- 项目是什么
- 属于什么类型
- 当前阶段是什么
- 本轮目标是什么

### Step 2：读模板
按项目类型读取：
- 微信小程序：`templates/template-weapp.md`
- 移动端 App：`templates/template-mobile-app.md`
- Web：`templates/template-web-app.md`

### Step 3：建图
先画出：
- 模块
- 页面/入口
- 数据流
- 主流程
- P0

### Step 4：抓关键用例
不要一上来分析全部用例。  
先抓：
- P0
- 主流程
- 高风险删除/保存/恢复/提交
- 状态变化场景

### Step 5：决定工作模式
- 能执行命令：走“可执行自检”
- 不能执行命令：走“静态自检”

### Step 6：取证
可执行模式优先顺序：
1. 构建
2. 启动
3. 自动化测试
4. 关键路径验证
5. 日志/截图/报告归档

静态模式优先顺序：
1. 代码路径
2. 配置与路由
3. 现有截图/日志/报告
4. 标注缺证据项

### Step 7：逐条判定
对每条关键用例输出：
- 编号
- 模块
- 目标
- 证据
- 状态
- 缺口
- 下一步

### Step 8：总门禁
严格按 `TEST_GATE.md` 汇总：
- PASS
- FAIL
- BLOCKED
- NEEDS_HUMAN_CHECK

## 三、推荐做法
- 先给出“本轮范围”，避免把整库所有内容都混在一起
- 先打 P0，再覆盖 P1/P2
- 先确认主流程，再看优化项
- 对“不能确定”的情况保守判定

## 四、常见错误
- 看到页面文件就说页面通过
- 看到存储逻辑就说数据持久化通过
- 没跑命令却写“构建通过”
- 没截图没日志却写“流程跑通”
- 把“待人工确认”隐藏掉

## 五、推荐最终产物
至少产出：
1. Markdown 自检报告
2. CSV 证据矩阵
3. JSON 摘要
