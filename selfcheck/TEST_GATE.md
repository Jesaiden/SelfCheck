# TEST GATE（统一测试门禁规则）

本文件定义什么情况下允许通过，什么情况下必须阻断。

## 一、总状态定义

### PASS
只有在以下条件同时满足时，才允许给出 PASS：
1. 全部 P0 用例已完成验证
2. 全部 P0 用例状态均为 PASS
3. 主闭环流程为 PASS
4. 不存在未解释的高风险 FAIL
5. 不存在阻断性的 BLOCKED
6. 关键证据链完整，且不依赖单独 C 级证据
7. `NEEDS_HUMAN_CHECK` 仅限已被项目配置允许的主观体验项，且不影响上线门禁

### FAIL
存在以下任一情况即判 FAIL：
1. 任一 P0 明确失败
2. 主流程闭环失败
3. 数据丢失、状态错乱、崩溃、严重逻辑错误
4. 保存/删除/覆盖/提交等高风险操作错误
5. 当前缺陷在修复前不建议提测或发布

### BLOCKED
存在以下任一情况即判 BLOCKED：
1. 缺少关键环境
2. 缺少关键账号/权限/依赖
3. 缺少测试用例或 PRD，导致关键项无法判定
4. 无法运行构建/启动/测试，导致主流程验证无法完成
5. 真机 / 模拟器 / 设备资源缺失导致关键结论无法得到

### NEEDS_HUMAN_CHECK
存在以下任一情况即判 NEEDS_HUMAN_CHECK：
1. 当前只能做到静态或半自动验证
2. 文案、视觉、动效、细微交互、真机手感等仍需人工确认
3. 兼容性、滚动手感、输入体验等不适合由当前代理单独定论
4. 项目配置明确要求人工兜底的项目仍未补测

---

## 二、P0 默认范围
除非项目配置另有说明，以下默认属于 P0：
- 新用户首次关键入口
- 主业务闭环
- 保存 / 提交 / 删除 / 覆盖
- 登录态 / 本地状态 / 权限状态变化
- 退出重进 / 中断恢复
- 历史查看 / 结果回显 / 关键数据持久化
- 支付 / 下单 / 发布 / 导出 等关键动作（若项目存在）

---

## 三、证据分级

### A 级证据（强证据）
- 自动化测试通过记录
- 构建成功 / 启动成功 / 命令执行成功
- 真机 / 模拟器关键页面截图或录屏
- 与用例直接对应的日志输出
- 可重复复现的闭环操作结果

### B 级证据（中证据）
- 部分运行成功结果
- 历史报告
- 局部可运行验证
- 部分闭环但仍缺最后一步的结果

### C 级证据（弱证据）
- 静态代码推断
- 页面存在
- 文案存在
- 开发者口头说明

### 分级规则
- PASS 至少需要 A 级，或足够强的 B 级组合
- 单独 C 级不能支撑 PASS
- P0 原则上不能只靠 B 级，更不能只靠 C 级
- 对主流程闭环，优先要求 A 级证据

---

## 四、复测门禁
修复后复测时，必须回答：
1. 哪些 FAIL 已修复？
2. 哪些 BLOCKED 已解除？
3. 哪些 NEEDS_HUMAN_CHECK 仍保留？
4. 是否重新覆盖到所有受影响 P0？
5. 总门禁是否发生变化？

### 允许局部复测的条件
- 缺陷影响面清晰
- 变更模块明确
- P0 影响范围可界定
- 已列出回归范围

### 不允许只做局部复测的情况
- 涉及主流程核心状态机
- 涉及保存 / 删除 / 登录态 / 本地存储
- 涉及多模块共享数据结构
- 涉及全局路由 / 生命周期 / 初始化逻辑

---

## 五、门禁输出必须包含
- 总状态
- P0 汇总
- 主流程结果
- FAIL 清单
- BLOCKED 清单
- NEEDS_HUMAN_CHECK 清单
- 复测建议
- 人工补测建议

---

## 六、禁止降级通过
以下情况一律禁止“勉强 PASS”：
- P0 未验证但主观判断“问题不大”
- 主流程没跑通但代码看起来正确
- 缺少证据却用模糊表述掩盖
- 只做静态审查却写成“测试通过”
